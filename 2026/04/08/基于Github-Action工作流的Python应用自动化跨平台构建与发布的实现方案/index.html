基于Github Action工作流的Python应用自动化跨平台构建与发布的实现方案

前言

Python无法类似Java实现“一处打包，到处运行”，需要使用CI/CD（如GitHub Actions、GitLab CI）在Windows、Linux、macOS上分别执行打包命令，生成各自平台的可执行文件。遂有此文。

正文

笔者之前曾有相关经验，将Github Action工作流的基础知识记录在了以下链接中，在此不再赘述。

https://waltyoung.github.io/2025/05/20/%E5%9F%BA%E4%BA%8EGithub-Action%E5%B7%A5%E4%BD%9C%E6%B5%81%E7%9A%84Python%E5%BA%94%E7%94%A8%E8%87%AA%E5%8A%A8%E5%8C%96%E6%9E%84%E5%BB%BA%E4%B8%8E%E5%8F%91%E5%B8%83%E5%AE%9E%E8%B7%B5/

需要强调的是，Github Action工作流文件路径为.github/workflows/[filename].yml，其中文件名可以自定义。为了可读性，文件名一般为build.yml或release.yml

以下是实现Python应用自动化跨平台构建与发布的Github Action工作流文件，支持以下功能：

  • 支持在Windows、Linux、macOS上分别生成各自平台的可执行文件
  • 使用Nuitka进行打包Python应用
  • 支持在推送一个形如v1.0.0的Git标签触发自动化构建：git tag v1.0.0 && git push --tags
  • 支持手动触发*(但手动触发的工作流不会发布Release)*

注意事项

  • 如果使用git push --tags创建Release失败，则需要进入仓库的 Settings → Actions → General → Workflow permissions，确保选择Read and write permissions
  • macOS代码签名：Nuitka生成的macOS可执行文件未签名，运行时会提示“无法验证开发者”。用户可通过xattr -d com.apple.quarantine myapp-macos解除限制。
  • Linux依赖：默认使用静态链接，但某些库（如libpython）仍需在目标系统存在。

遇到的问题与解决方案

在实际配置与运行上述工作流的过程中，笔者遇到了若干典型问题，现将问题现象及解决方法记录如下，供读者参考。

Windows 构建时 Nuitka 请求下载 Dependency Walker

现象：
在 Windows 运行器上执行python -m nuitka --standalone --onefile时，Nuitka 输出以下信息并退出：

由于 GitHub Actions 环境是非交互式的，Nuitka 默认选择了No，导致构建失败。

原因：
Nuitka 在 Windows 上分析扩展模块依赖时需要 Dependency Walker 工具（depends.exe），该工具并非 Nuitka 内置，需要首次使用时下载。非交互式终端无法响应下载提示。

解决方法：

Windows 运行器上 Shell 语法不兼容

现象：
在安装依赖步骤中使用if [ -f requirements.txt ]; then pip install -r requirements.txt; fi时，Windows 运行器（默认 PowerShell）报错：

原因：
GitHub Actions 的 Windows 运行器默认使用 PowerShell，而上述命令是 Bash 语法。PowerShell 无法识别。

解决方法：

提示nuitka: command not found

现象：
在 Windows 运行器上直接使用nuitka命令时，提示command not found，尽管 pip install nuitka已经成功执行。

原因：
Nuitka 安装后，其可执行文件位于 Python 安装目录的Scripts子目录（Windows）或bin子目录（Linux/macOS）。在某些运行器环境下，该路径未被自动添加到 PATH 环境变量中。

解决方法：
统一使用python -m nuitka代替直接调用nuitka命令。这样会通过 Python 模块方式运行，无需依赖PATH设置，且在所有平台上行为一致。

Linux 可执行文件因 glibc 版本过低无法运行

现象：
在ubuntu-latest（当前为 Ubuntu 24.04，glibc 2.39）上构建的 Linux 可执行文件，部署到较老的服务器（如 Ubuntu 22.04，glibc 2.35）上运行时，报错：

原因：
Nuitka 生成的可执行文件会动态链接构建环境中的 glibc。新版 Ubuntu 使用的 glibc 版本高于目标服务器，导致低版本系统无法解析新版本符号。

解决方法：
将 Linux 构建的运行环境改为较老但仍在维护的 Ubuntu 版本，例如ubuntu-22.04（glibc 2.35）或ubuntu-20.04（glibc 2.31）。大多数生产环境仍可兼容这些版本。修改矩阵定义：
