基于DataX的自Trino到MySQL的大规模数据同步工程实践(一)

前言

笔者接到一个需求——从Trino（原 PrestoSQL）组装AI所需的JSON数据。但笔者发现在Trino进行SQL查询需要2min左右的时间，故计划将数据从Trino同步到MySQL数据库，再在MySQL数据库中查询，遂有此文。

正文

Trino

Trino(起源于Facebook2012年开源的Presto项目)并不是一个传统意义上的“数据库”，而是一个分布式SQL查询引擎。

Trino主要特点

  1. 无存储层
    Trino 本身不管理任何数据存储，也不存储自己的数据。它只负责解析 SQL、生成执行计划，并协调工作节点从各个数据源读取数据。

  2. 联邦查询（跨源联合）
    通过一套 SQL 语法，可以同时查询 MySQL、PostgreSQL、Kafka、Hive、Iceberg、Elasticsearch、MongoDB 等多种数据源，并做关联分析。

  3. 高性能与低延迟
    基于内存并行处理架构，利用多核 CPU 和分布式节点进行大规模并行处理（MPP），适合亚秒级到分钟级的交互式分析场景。

  4. 标准 SQL 支持
    支持 ANSI SQL 的大部分特性（JOIN、聚合、窗口函数、子查询等），并兼容多种客户端工具（DBeaver）。

  5. 扩展性与弹性
    节点可水平扩展，能根据负载动态增减工作节点（通常配合容器编排如 Kubernetes 使用）。

Trino与数据库的区别

特性 Trino 传统数据库（如 MySQL、PostgreSQL）
数据存储 无，依赖外部数据源 自带存储引擎，管理数据文件
数据写入 不支持（或有限支持 INSERT/CTAS） 完整的 CRUD 操作
索引/事务 无事务，无自有索引 支持事务（ACID）、二级索引
适用场景 多源联邦查询、大数据分析、BI 报表 OLTP 或小规模 OLAP

Trino常见使用场景

  • 数据湖分析（Hive、Iceberg、Delta Lake 上的数据）
  • 多源异构数据的临时探索式查询
  • 实时数据流与历史数据的联合分析（Kafka + HDFS）
  • 替换传统的 ETL 流程，直接对原始数据查询

解决方案

笔者选择采用阿里巴巴开源的轻量级ETL工具——DataX，因其支持丰富的异构数据源，性能高、配置简单

需要提醒读者的是，截止到笔者落地该解决方案(2026年4月)，DataX已有3年未更新，最近一次更新在2023年9月

安装 DataX

DataX是纯Java应用，只需下载解压即可。

DataX项目链接：https://github.com/alibaba/DataX

DataX下载链接：https://datax-opensource.oss-cn-hangzhou.aliyuncs.com/202308/datax.tar.gz

安装 JDBC 驱动

DataX的结构相对较易理解，由reader负责从源数据库读取，由writer负责向目标数据库写入

故在笔者的任务中需要trinoreader插件从Trino中读取数据和mysqlwriter插件向MySQL中写入数据，但是因为DataX没有正式支持Trino或Presto的专用插件，所以笔者改用rdbmsreader（通用关系数据库读取器）代替trinoreader。

DataX的rdbmsreader插件和mysqlwriter插件需要对应的JDBC驱动：

  • Trino JDBC 驱动下载地址：https://trino.io/download
  • MySQL JDBC 驱动下载地址：https://dev.mysql.com/downloads/connector/j/

将Trino JDBC 驱动放到对应的reader目录(datax/plugin/reader/rdbmsreader/lib/)下，并设置权限为755

将MySQL JDBC 驱动放到对应的writer目录(datax/plugin/writer/mysqlwriter/lib/目录下，并设置权限为755

注意：不要重复放置不同版本的驱动，以免冲突。

未找到 JDBC 驱动（No suitable driver found）

现象
java.sql.SQLException: No suitable driver found for jdbc:trino://...

原因
使用 rdbmsreader插件后 libs 目录下没有放置 Trino JDBC 驱动。

解决方法

  • 下载 Trino JDBC 驱动（trino-jdbc-*.jar）和 MySQL JDBC 驱动（mysql-connector-java-*.jar）。
  • 将 Trino 驱动放入 datax/plugin/reader/rdbmsreader/libs/。
  • 将 MySQL 驱动放入 datax/plugin/writer/mysqlwriter/libs/。
  • 设置文件权限为 755。

数值类型转换溢出（INT / BIGINT 范围不足）

现象
Cannot cast '9000015936' to INT
Cannot cast '16786729416200931469' to BIGINT

原因
Trino 中存储的数字字符串长度超过目标整数类型的最大值（INT 最大 21 亿，BIGINT 最大约 9.22e18）。

解决方法

  • 对于超过 INT 但未超过 BIGINT 的值，使用 CAST(col AS BIGINT)。
  • 对于超过 BIGINT 的超大整数（如 19 位以上），改用 CAST(col AS DECIMAL(20,0))，并相应修改 MySQL 目标列为 DECIMAL(20,0)。
  • 如果不需要数值计算，可直接保留为字符串类型。

现象
Query exceeded the maximum execution time limit of 20.00m

原因
Trino 集群或 catalog 层面默认查询超时较短（20分钟），而全表扫描数据量巨大，无法在时限内完成。

  • 如果 splitPk 不可用（如分片列是字符串类型导致 SQL 语法错误），则手动使用 where 条件分批同步，例如 where = "id >= 0 AND id < 1000000"。

内存溢出（OutOfMemoryError: Java heap space）

现象
java.lang.OutOfMemoryError: Java heap space，并生成 .hprof 堆转储文件。

原因

  • fetchSize 过大，且表中存在长字段（如 xml_cont、history_summary），导致单个通道缓存的数据量超过 JVM 堆内存。
  • DataX 默认 JVM 堆内存（通常为1~2GB）不足。

解决方法

  • 降低 fetchSize（例如从 2048 降到 200~500）。
  • 降低并发 channel 数量（例如从 4 降到 1，或根据内存调整）。
  • 增加 DataX 启动堆内存：修改 bin/datax.py 中的 DEFAULT_JVM，例如 -Xms2g -Xmx4g。
  • 如果表中大字段非必需，可在 column 中省略，或使用 SUBSTR(col, 1, 1000) 截断。

脏数据（Data too long for column）

现象
日志中出现 Data truncation: Data too long for column 'operation_level' at row 1，记录被跳过。

原因
源字段的实际长度超过 MySQL 目标字段定义的长度（例如 VARCHAR(10) 接收了 20 个字符）。

解决方法

  • 查询 Trino 中该字段的最大长度：SELECT MAX(LENGTH(column)) FROM table。
  • 根据结果扩大 MySQL 字段长度：ALTER TABLE ... MODIFY COLUMN ... VARCHAR(新长度)。

性能调优

并发通道与分片键（channel + splitPk）

DataX通过channel控制并发线程数，每个线程独立处理一部分数据。**channel 只有与 splitPk 配合才能真正实现并行读取**。

  • **channel**：设置并发度，默认值为 1。建议根据 DataX 机器 CPU 核数、源端和目标端的承受能力调整，通常取4~8。
    配置示例：

  • **splitPk**：指定一个整数类型的主键或自增列，DataX 会自动将该列的值域切分成 channel份，每个通道只查询一部分数据（如 WHERE id BETWEEN 1 AND 10000）。
    要求：该列必须是整型（INT / BIGINT），且分布相对均匀，建议有索引。

注意：如果分片列是 VARCHAR（即使存的是数字），splitPk 会生成错误的 SQL 语法（如 '3|T)G'4' <= rec_sn），此时不可使用 splitPk，只能退而使用where手动分页。

批量大小（fetchSize``batchSize）

  • fetchSize（Reader 端）：每次从Trino拉取的行数。

    • 默认值：1000
    • 增大可以减少网络往返次数，提高读取效率。
    • 注意：如果单行数据较大（如包含 TEXT/JSON/XML 字段），过大的 fetchSize 会导致 DataX 内存溢出（OOM）。建议根据行大小调整，例如普通行（每行几百字节）可设为5000~10000；存在长字段时降到200~500。

  • batchSize（Writer 端）：每次向MySQL提交的行数。

    • 默认值：1024
    • 增大可以减少 MySQL 事务提交次数，显著提升写入吞吐量。
    • 注意：batchSize 过大可能导致单条 SQL 语句长度超过 MySQL max_allowed_packet 限制（默认 4M）。一般建议1000~5000，若行很小可适当增大。

MySQL写入优化：rewriteBatchedStatements=true

MySQL JDBC 驱动默认的 executeBatch 不会真正合并 SQL，而是逐条发送。添加参数 rewriteBatchedStatements=true 后，驱动会将多条INSERTSQL语句重写为一条多值的INSERT语句，可提升写入性能 5~10 倍。

    • SET SESSION hive.optimized_partition_update_skip = true（如底层是 Hive）：跳过不必要的分区更新，提高读取速度。
