SQL-游戏玩法分析IV

原题链接:https://leetcode.cn/problems/game-play-analysis-iv/description/?envType=study-plan-v2&envId=sql-free-50

Problem

题目描述

Column Name Type
player_id int
device_id int
event_date date
games_played int

（player_id，event_date）是此表的主键（具有唯一值的列的组合）。
这张表显示了某些游戏的玩家的活动情况。
每一行是一个玩家的记录，他在某一天使用某个设备注销之前登录并玩了很多游戏（可能是 0）。

编写解决方案，报告在首次登录的第二天再次登录的玩家的 比率，四舍五入到小数点后两位。换句话说，你需要计算从首次登录日期开始至少连续两天登录的玩家的数量，然后除以玩家总数。

输入例子

player_id device_id event_date games_played
1 2 2016-03-01 5
1 2 2016-03-02 6
2 3 2017-06-25 1
3 1 2016-03-02 0
3 4 2018-07-03 5

输出例子

fraction
0.33

题目难度

部分代码解释

上述SQL语句使用嵌套查询，包括一层子查询。

分母使用(SELECT COUNT(DISTINCT player_id) FROM Activity)标量子查询，以避免外部SQL干扰。

子查询生成的临时表必须有别名，如本题的A1。

ROUND(number, decimal_places)将数值四舍五入到指定小数位数

  • number：需要格式化的数值
  • decimal_places：保留的小数位数

DATEDIFF(end_date, start_date)计算两个日期之间的天数差。注意参数顺序

  • end_date：结束日期
  • start_date：开始日期

窗口函数LEAD(column, offset, default_value) OVER(...)获取分区内当前行之后的某行数据

  • column：目标列
  • offset（可选参数）：向后偏移的行数（默认为 1）
  • default_value（可选参数）：当后续没有足够行时（如分区末尾），返回的默认值。若未指定，默认返回NULL。（本题中未指定默认值，故最后一行的next_event_date的值为 NULL，但这个值会被外层查询的WHERE子句过滤，算是一个潜在问题）
  • OVER(…)：定义窗口范围。LEAD()函数后必须接OVER()函数

窗口函数LAG(column, offset, default_value) OVER(...)

  • column：目标列
  • offset（可选参数）：向前偏移的行数（默认为 1）
  • default_value（可选参数）：当之前没有足够行时（如分区开始），返回的默认值。若未指定，默认返回NULL
  • OVER(…)：定义窗口范围。LAG()函数后必须接OVER()函数

窗口子句OVER(PARTITION BY column_1 ORDER BY column_2)定义窗口函数的分区和排序规则，确保窗口函数正确计算。

  • PARTITION BY column_1：按某列分区（本题中为 player_id）。
  • ORDER BY column_2：按某列排序（本题中为 event_date）。
    OVER()函数可不定义窗口函数的分区和排序规则，即不写PARTITION BY和ORDER BY。在小数据量时，可能是正确的，但大数据量时易报错。故OVER()函数建议写PARTITION BY和ORDER BY。
